## 1. Overview

### 1.1 Design Goals

본 사업은 시스템 구현이 아닌 **발주처 전달용 V2.0 산출물 3종 작성**이 목표이다. Design 단계의 목표는:

1. 각 산출물의 **섹션별 콘텐츠 매핑**을 사전 정의하여 작성 흐름을 명확화
2. **단일 진실 공급원(Single Source of Truth)** 을 지정하여 세 산출물 간 수치·표현 일관성 확보
3. **34건 협의 결정 사항**이 어느 산출물의 어느 섹션에 반영될지 명시적으로 지정
4. 작성 후 **design-validator 정합성 검증** 기준을 사전 정의

### 1.2 Design Principles

- **단일 진실 공급원**: `feature-list.md`(MECE 기능 목록)와 `comparison-analysis.md`(비교 분석)를 참조 기준으로 삼고, 세 산출물은 이를 일관되게 재구성
- **추적성 우선**: 모든 결정 사항에 (2026-05-07 미팅) / (2026-05-11 미팅) 출처를 명시
- **재사용 가능 구조**: 세 산출물의 부록은 동일한 기능 목록을 공유 (중복 작성 금지)
- **차별화된 목적**: RFI 답변서(설득·신뢰), 사업계획서(범위·일정), 견적서(비용·산식) 각각의 목적에 맞는 톤·구성 유지

## 2. Architecture (산출물 간 관계도)

### 2.1 산출물 간 의존성

```
                  ┌────────────────────────────────────────┐

                  │  • followup-questions.md (협의 결과)    │

                  └────────────┬───────────────────────────┘

            ↓                  ↓                  ↓
   ┌────────────────┐  ┌────────────────┐  ┌────────────────┐
   │ RFI 답변서      │  │ 사업계획서      │  │ 견적서          │
   │ uscp-rfi-      │  │ uscp-proposal- │  │ estimate-      │
```

### 2.2 일관성 확보를 위한 핵심 수치·표현 (단일 진실)

| 항목 | V2.0 공통 값 |
|------|------------|
| 대상 지역 | 5개 (대전·공주·예산·천안·세종) |
| 양교 협력 | 공주대(주관) + 충남대(단순 협력 기관) |
| 사업 단계 | 1차년도(구축·시범운영) + 2027년 이후(고도화) |
| 운영 모델 | 로그인 회원 제보 + 게이트키핑 (바텀업 비로그인 제보 폐기) |
| 의제 단계 | 6단계 (제보 → 검토중 → 공개등록 → 멘토배정 → 처리중 → 해결완료) |
| 사용자 역할 | 시민 회원 + 지역사회특화센터 운영자(단일 역할) + 멘토 + 학생팀 |
| 알림 채널 | 이메일 단일 |
| CMS 범위 | 소개 정적 HTML / 공지·이벤트·약관 WYSIWYG / 단순 배너 관리 |
| 멘토 매칭 | 관리자가 가입자 중 선정 + 수동 매칭 + 통보형 |
| 견적 범위 | 5,000~8,500만 원 (시나리오 B', VAT 포함) |
| 트래픽 | 최대 100명 미미한 수준 |
| AI 자동화 | 미적용 |
| 양교 SSO | 연동 없음 |

### 2.3 콘텐츠 흐름 (작성 순서)

세 산출물의 작성 순서는 **데이터 우선**으로 진행한다:

```
① 견적서 (estimate-v2.md) — 수치·산식 확정 우선
       ↓
② 사업계획서 (uscp-proposal-v2.md) — 견적 근거가 되는 범위·기능 명세
       ↓
③ RFI 답변서 (uscp-rfi-response-v2.md) — 사업계획서·견적서를 발주처 관점에서 재구성
```

작성 우선순위 사유: 견적서가 가장 정량적이고 후속 두 문서의 산식·일정·기능 범위 기준이 된다. 견적서를 먼저 확정하면 사업계획서·RFI 답변서의 표현·수치를 일관되게 도출할 수 있다.

## 3. Data Model (산출물 콘텐츠 모델)

### 3.1 협의 결정 사항 데이터 구조 (34건)

```yaml
협의 결정 사항:
  2026-05-07 사전 협의 (3건):
    - 권한 관리 기능 적용
    - 감사 로그 적용
    - 개인정보 동의 약관 적용

  PDF 명시 해결 (3건):
    - 천안 = 충남대 천안캠퍼스 (PDF p.2)
    - 학생팀 = 사회혁신가 양성과정 (PDF p.3)
    - 산출물 공개 (PDF p.18)

  2026-05-11 1차 미팅 (28건):
    운영 모델 변경: 1건
    알림: 5건
    개인정보: 4건
    CMS: 4건
    멘토단·학생팀: 5건
    양교 협력: 4건
    권한: 4건
    비기능: 2건
    투표 트랙: 3건 (단순 라벨 통합)
    데이터 소유권: 1건
```

### 3.2 기능 ID 체계 (feature-list.md 기준)

각 산출물의 부록에서 동일한 기능 ID 체계를 사용한다:

```
HOM-*    : 홈 (12개)
ABT-*    : USCP 소개 (5개, ABT-04 신규 양교 MOU 안내)
ISS-*    : 지역문제 광장 (7개)
LL-*     : 리빙랩 공개 (6개)
NET-*    : 협력 네트워크 (5개)
PERF-*   : 성과자료 (5개)
SUB-*    : 회원 제보 (10개 중 5개 제거됨)
VOTE-*   : 공감 투표 (7개 중 1개 자동 해결)
CMT-*    : 댓글 (6개 중 1개 해결)
NOTIF-*  : 알림 (6개 중 2개 해결)
AUTH-*   : 인증 (11+1개, AUTH-12 시민 회원가입 신규)
GATE-*   : 게이트키핑 (13개 중 1개 해결)
LLM-*    : 리빙랩 관리 (12개)
MNT-*    : 멘토단 운영 (10개 중 1개 해결)
NETM-*   : 협력 네트워크 관리 (8개)
PERFM-*  : 성과자료 관리 (7개)
CMS-*    : 콘텐츠 관리 (11개 중 1개 제거, 1개 신규)
PERM-*   : 권한 (6개 중 3개 단순화)
AUD-*    : 감사 (6개)
PRIV-*   : 개인정보 (4개)
INT-*    : 내부 시스템 (45개)
BG-*     : 백그라운드 (18개)
NFR-*    : 비기능 (27개)
```

### 3.3 산출물별 콘텐츠 요소

```typescript
interface RFIResponse {
  submissionConfirmation: { 수신·제출자·대표자 };
  companyOverview: { 회사·강점 };
  rfiUnderstanding: { 사업 배경·핵심 요구·도전 과제 };
  proposalDirection: { 철학·단계 전략·데이터 흐름 };
  platformIA: { 5개 지역·6단계·사용자 구조 };
  detailedScope: { 1차년도·2027년 이후 };
  techEnvironment: { 9개 기술 결정 };
  capabilities: { 유사 실적·팀 구성·방법론 };
  schedule: { 마일스톤·타임라인 };
  cost: { Indicative 5,000~8,500만 원 };
  risks: { 6+가지 리스크 };
  maintenance: { 무상 1년 + 유상 SLA };
  deliverables: { 산출물 목록 };
  differentiators: { 차별점 5개 };
  appendix: { A: 기능 / B: 결정 사항 / C: 실적 양식 };
}

interface BusinessProposal {
  overview: { 목적·배경·관점 (실무자 시점) };
  IA: { 5개 영역·6단계·사용자 구조 };
  scope: { 1차년도·2027년 이후 };
  techEnvironment: { 동일 };
  schedule: { 1차+고도화 };
  appendix: { 통합 기능 목록 };
}

interface Estimate {
  overview: { 견적 가정·범위 };
  scenarioBPrime: { 5,000~8,500만 원 };
  stageBreakdown: { 분석설계·개발·시범운영·고도화 };
  variationItems: { 11개 변동 항목 };
  optionalVariables: { LLM·SSO·CAPTCHA };
  maintenancePolicy: { 무상·유상 };
}
```
