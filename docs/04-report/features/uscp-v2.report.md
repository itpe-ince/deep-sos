### 2. 개선 점 (Areas for Improvement)

#### 2.1 **표면 표기 정확성의 사전 검증**
- **이슈**: 부록 A 헤더의 카운트 오류 (28건 vs 33행)
- **근본 원인**: 28건의 핵심 결정이 실제 본문에서 33개 세부 항목으로 세분화되는 과정이 설계 단계에서 명시되지 않음
- **개선안**: Design 단계에서 "핵심 결정 수 vs 세부 항목 수"의 변환 규칙을 명시하고, 작성 후 체크리스트로 검증
- **예방**: `§3.1 협의 결정 사항 데이터 구조`에서 "각 결정이 1~N개 세부 항목으로 확장됨"을 명시

#### 2.2 **신규 항목(AUTH-12)의 조기 발견**
- **이슈**: Design 단계에서 AUTH-12 신규 항목이 명시되었으나, Do 단계 작성 시 부록 A.1에서 누락
- **근본 원인**: 신규 항목이 기존 항목 목록(AUTH-01~08)과 분리되어 별도 섹션에 기록되어 있어 일관성 검증이 누락
- **개선안**: Design 단계에서 "신규 항목 목록"을 명시적인 체크리스트로 작성
  ```
  신규 항목:
  - AUTH-12 시민 회원가입 (1차 미팅)
  - ABT-04 양교 MOU 안내 (1차 미팅)
  - CMS-11 이벤트 공지 WYSIWYG (1차 미팅)
  ```

#### 2.3 **산식 일관성의 자동화 검증**
- **이슈**: RFI §9.3 AI 가산 범위가 estimate §5와 불일치 (600~1,800만 vs 1,400~2,200만)
- **근본 원인**: 산식이 여러 문서에 산재되어 있어 수동 일관성 검증이 누락
- **개선안**: Design 단계에서 "산식 일관성 검증 표"를 명시
  ```
  | 항목 | estimate | RFI | 불일치? |
  |------|---------|-----|---------|
  | AI 챗봇 | +600~1,000만 | ? | |
  ```

#### 2.4 **부록 일관성 규칙의 조기 결정**
- **이슈**: 해결된 기능의 취소선+✅ 표기가 AUTH-11에만 적용되고, 다른 기능(SUB·VOTE·CMT 등)에는 누락
- **근본 원인**: Design §7.3에서 "취소선+✅로 표기"를 규칙으로 명시했으나, "어느 범위까지 적용"할지는 명시하지 않음
- **개선안**: Design 단계에서 "부록 A 처리 전략"을 명시
  ```
  해결·제거 항목 처리:
  - 1차 미팅 결정으로 해결: ~~취소선~~ + ✅ (필수)
  - 본 사업 범위 외: 단순 누락 또는 "(범위 외)" 표기 (선택)
  ```

#### 2.5 **발주처 제출 전 최종 보완 체크리스트 부재**
- **이슈**: 회사 정보·실적 플레이스홀더가 발주처 제출 직전 단계에서 발견
- **개선안**: Do 단계 완료 후 "발주처 제출 전 최종 체크리스트"를 생성
  ```
  [ ] 회사명·대표자·주소 실제 정보
  [ ] 유사 수행 실적 포트폴리오 입력
  [ ] 이메일·연락처 유효성 검증
  [ ] PDF 변환 후 레이아웃 확인
  ```

### 3. 학습 사항 (Lessons Learned)

#### 3.1 **PDF 명시 사항의 중요성**
**학습**: 발주처 제공 PDF(한혜진 계획안)에 명시된 세부 사항(천안 = 충남대 천안캠퍼스, 학생팀 = 사회혁신가 양성과정)이 협의 결정처럼 중요하며, 이를 "PDF 명시 해결 (3건)"으로 분류하여 명시적으로 추적해야 함.

**적용**: 향후 유사 사업에서는 발주처 제공 문서를 먼저 분석하여 명시 사항 목록을 작성하고, 협의 결정과 별도로 추적.

#### 3.2 **1차 미팅 결정의 세분화 필요성**
**학습**: "멘토단 관리" 1개 결정이 실제로는 5개 세부 항목(멘토 풀·등록·매칭·수락·학생팀)으로 세분화됨. 이를 미리 파악하지 않으면 부록 헤더와 본문 간 카운트 불일치 발생.

**적용**: 1차 미팅 후 결정 사항을 즉시 세분화하여 "28건 결정 → 33개 세부 항목"의 매핑 표를 작성. Plan 단계에서 이 표를 기준으로 작성.

#### 3.3 **단일 진실 공급원(SSOT) 개념의 실질적 가치**
**학습**: feature-list.md(기능 ID 219건)와 comparison-analysis.md(변경 분석)를 SSOT로 지정하면, 세 산출물 간 일관성 검증이 명확해짐. 특히 기능 ID(HOM·ABT·ISS·AUTH 등)를 일관되게 사용하면 부록 검증이 쉬워짐.

**적용**: 향후 다중 산출물 사업에서는 처음부터 SSOT를 명시하고, 모든 문서가 SSOT를 참조하도록 구조화. 이는 일관성 검증 도구(design-validator)의 정확도를 크게 향상.

#### 3.4 **작성 순서의 전략적 중요성 (역순)**
**학습**: 일반적으로 RFI → 사업계획서 → 견적 순서로 생각하지만, 본 사업에서는 견적 → 사업계획서 → RFI 순서로 진행함. 이는 견적(가장 정량적)을 먼저 확정하면, 사업계획서와 RFI의 수치와 표현이 자동으로 일관되기 때문.

**적용**: 향후 유사 사업에서는 "데이터 우선 작성 순서"를 채택. 즉, 가장 정량적·제약이 많은 문서부터 작성하여, 후속 문서의 작성이 쉽고 일관되도록 구조화.

#### 3.5 **검증 도구의 이원화 가치**
**학습**: gap-detector(사업 본질 충족도 99%)와 design-validator(표면 정확성 88→96%)의 점수가 다르면, 단순히 "평균 점수"만 보는 것이 아니라 "어느 영역에서 차이가 발생했는가"를 분석해야 함. 이를 통해 Critical/Major/Minor 이슈를 구분하고, 수정 우선순위를 결정할 수 있음.

**적용**: 향후 검증 단계에서는 gap-detector와 design-validator의 점수를 따로 보고, 차이가 크면 원인 분석 단계를 추가 삽입. 이를 통해 "사업 본질은 맞으나 표면 표기가 부정확"한 경우를 조기에 발견 가능.

#### 3.6 **발주처 제출 전 최종 보완의 체계화**
**학습**: PDCA Check 단계에서 Match Rate 97%를 달성했지만, 발주처 제출 직전에 "회사 정보·실적 플레이스홀더 → 실제 데이터"로 변환해야 함. 이는 PDCA 사이클 외 작업이므로, 보고서에서 명시적으로 "발주처 제출 직전 체크리스트"로 기록해야 함.

**적용**: 보고서의 "발주처 전달 패키지 안내" 섹션에서 3개 필수 보완 항목(회사 정보·실적·표기)을 명시하고, 소요시간을 추정. 이를 통해 최종 제출까지의 남은 작업을 명확히 함.
