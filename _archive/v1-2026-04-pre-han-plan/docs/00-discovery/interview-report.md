## 1. 인터뷰 목적

2025 글로컬대학 본지정 사업 실행계획서(§4-3)에 명시된 "온라인 사회공헌 플랫폼(USCP) 개발 및 운영"의 **구체적인 구현 범위와 깊이**를 확정하기 위해 인터뷰를 진행하였습니다.

실행계획서에는 리빙랩 운영, ESG 활동, 봉사활동 관리, 글로벌 확산 등 다양한 사업이 언급되어 있으나, 이 중 플랫폼이 실제로 어디까지 담당해야 하는지는 해석에 따라 크게 달라질 수 있습니다. 본 인터뷰를 통해 **발주처의 실제 운영 상황과 현실적인 제약조건**을 반영한 확정된 범위를 도출하였습니다.

## 2. 플랫폼 개요

### 2.1 플랫폼의 정체성

본 플랫폼은 다음 두 가지 성격을 결합한 시스템입니다.

- **소통 홈페이지**: 대학·지자체·시민에게 사업 소개, 성과 공유, 참여 안내를 제공하는 대외 홍보 채널
- **리빙랩 프로젝트 관리 시스템**: 지역사회문제 해결을 위한 리빙랩 프로젝트의 참여자, 일정, 예산, 결과물을 체계적으로 관리하는 내부 운영 도구

초기 요구사항(v1 답변)에서는 봉사활동, ESG, KPI 대시보드 등을 포함한 "종합 플랫폼(⑤)"으로 논의되었으나, 인터뷰 과정에서 **운영 인력(1인), 예산 규모, 실질적 활용도**를 고려하여 핵심 기능에 집중하는 방향으로 범위가 조정되었습니다.

### 2.2 핵심 사용자

인터뷰에서 확인된 플랫폼 사용자 구조는 다음과 같습니다.

| 역할 | 대상 | 플랫폼에서의 활동 | 비고 |
|------|------|------------------|------|
| **관리자** | 지역사회특화센터 실무자 | 플랫폼 전체 운영: 지역 문제 등록, 프로젝트 생성/관리, 소개 페이지 편집, 기관/MOU 관리, 성공사례 등록 | **1인 운영** — 가장 핵심적으로 매일 사용하는 사용자 |
| **참여자** | 교수, 학생 | 소속 프로젝트의 산출물(결과물) 업로드, 팀 정보 확인 | 직접 프로젝트를 생성하거나 관리하지는 않음 |
| **열람자** | 지자체, 기업, 시민 등 | 공개 페이지 열람: 플랫폼 소개, 진행 중인 프로젝트, 성공사례 조회 | 별도 회원가입 없이 접근 가능한 공개 영역 |

**핵심 포인트**: 기존 논의(v1)에서는 교수/학생과 시민이 직접 문제를 제보하고 프로젝트에 참여 신청하는 **바텀업(bottom-up) 방식**이 고려되었으나, 인터뷰 결과 센터 관리자가 사업을 기획하고 팀을 구성하는 **탑다운(top-down) 방식**으로 확정되었습니다.

### 2.3 운영 원칙

인터뷰에서 반복적으로 강조된 운영 원칙은 다음 네 가지입니다. 이 원칙들은 향후 모든 설계 및 구현 의사결정에 우선하여 적용됩니다.

**1) 1인 운영 체제**
지역사회특화센터 전담 인력 1명이 플랫폼 전체를 운영합니다. 타 글로컬대학 사례에서도 운영 인력이 1명인 경우가 대부분이며, 운영자의 부담을 최소화하는 설계가 필수적입니다. "관리자 기능도 자동으로 되면 좋겠다"는 의견이 있었으며, 수동 작업이 많은 기능은 의도적으로 배제하기로 하였습니다.

**2) 탑다운 운영 방식**
시민이 자유롭게 문제를 올리는 것이 아니라, 센터 담당자가 조사/기획한 후 플랫폼에 등록합니다. 사업의 방향은 "이런 사업을 할 거야, 제안을 해줘"라는 방식으로 위에서 아래로 내려오는 구조입니다.

**3) 산출물 중심 관리**
프로젝트의 진행 과정을 단계별로 모니터링/체크하는 것이 아니라, 팀이 내부적으로 프로젝트를 수행하고 **최종 결과물(산출물)을 플랫폼에 업로드하는 방식**으로 관리합니다. 중간 모니터링 기능을 넣으면 사용자와 운영자 모두의 업무 복잡도가 증가하므로 의도적으로 배제하였습니다.

**4) 간소화 우선**
기능을 많이 넣는 것보다 운영 효율을 높이는 것이 우선입니다. 이 원칙에 따라 봉사활동 관리, ESG/SDGs 성과 관리, 개인별 KPI, 온라인 협업 도구 등이 범위에서 제외되었습니다.

## 3. 기능 범위

### 3.1 포함 범위 (In Scope)

아래 기능들은 1차 구축 범위에 포함되며, 우선순위 순서대로 기술합니다.

#### (1) 리빙랩 프로젝트 관리 — 우선순위 1

플랫폼의 핵심 기능입니다. 리빙랩 5단계 프로세스(탐색 → 실행 → 개발 → 검증 → 활용)에 따라 프로젝트를 관리합니다.

- **운영 규모**: 3개 참여 그룹(학생, 지역시민, 교수)이 각각 약 10개 팀으로 참여. 연간 리빙랩 프로젝트 수는 매년 정해짐 (20~30개 수준)
- **관리 수준**: 참여자, 일정, 예산, 결과물 등 프로젝트 전체 정보를 관리하되, **플랫폼은 산출물 중심으로 관리**하고 프로세스 모니터링은 배제
- **팀 관리**: 팀당 인원수와 팀장 지정 방식은 아직 미정이나, 최소한의 정보만 관리하고 누가 팀장인지 식별 가능한 수준
- **메이커 리빙랩 3단계**(아이디어 → 실증 → 정책): 별도 프로세스가 아니라 5단계 안에 포함되는 프로젝트 분류(태그) 수준
- **사업화 연계**: 별도 관리 기능 없이, 프로젝트에 "사업화 연결됨" 정도의 표시만 있으면 충분
- **문서 양식 관리**: 관리자가 산출물 양식(템플릿)을 등록하면, 참여 학생들이 양식을 다운로드받아 작성 후 결과물을 업로드하는 흐름

#### (2) 지역 문제 관리 — 우선순위 2

지역사회의 문제를 플랫폼에 등록하고, 이를 기반으로 리빙랩 프로젝트와 연결하는 기능입니다.

- **등록 방식**: 센터 담당자가 조사한 후 **관리자만 등록** (시민 직접 등록 없음)
- **등록 후 프로세스**: 별도의 검토/승인 워크플로우 없이, 담당자가 결정하고 바로 등록
- **사례 공유**: 대전세종충남 리빙랩 네트워크 내에서 캠퍼스 간 사례를 서로 볼 수 있으면 되며, 필요 시 외부 링크 공유와 네트워크 연동도 지원

#### (3) 참여 기관 / MOU 관리 — 우선순위 3

리빙랩 프로젝트에 참여하는 지자체, 연구기관, 기업 등의 정보와 협약(MOU) 상태를 관리합니다.

- **기관 가입**: 기관 담당자가 플랫폼에 직접 가입하여 사용하지 않음. **관리자가 기관 정보를 등록하고 프로젝트에 이름만 연결**하는 방식
- **MOU 관리**: 어떤 기관과 어떤 협약이 체결되어 있는지를 플랫폼에서 확인할 수 있어야 함 (협약 제목, 기간, 상태 등)

#### (4) 콘텐츠 관리 (소개 페이지 편집) — 우선순위 4

플랫폼 소개, 참여 방법 안내, 캠퍼스 정보 등 대외 홍보용 정보 페이지를 관리자가 직접 편집할 수 있는 기능입니다.

- WYSIWYG(위지위그) 에디터를 통해 비개발자인 관리자가 직접 텍스트와 이미지를 편집하고 발행
- 대상 페이지: 플랫폼 소개(about), 참여 방법 안내(guide), 캠퍼스별 리빙랩, FAQ 등

#### (5) 성공사례 관리 — 우선순위 5

리빙랩 프로젝트의 성과를 정리하여 외부에 공개하는 기능입니다.

- 성공 사례를 외부에 공개하는 소개 페이지 형태로 제공
- 성과 기록을 아카이빙하여 축적해 놓고, 필요 시 공유/확산에 활용
- 지자체 정책에 반영된 경우, "이 프로젝트가 OO 정책에 반영되었다"는 기록을 남기는 수준

#### (6) 문서 양식 관리 — 우선순위 6

프로젝트 수행에 필요한 양식(템플릿)을 관리하고 배포하는 기능입니다.

- 관리자가 보고서 양식, 활동 기록 양식 등을 등록
- 참여 학생/교수가 필요한 양식을 다운로드받아 사용
- 작성 완료된 결과물을 다시 플랫폼에 업로드

#### (7) 지역교육 혁신 — 우선순위 7

공동 교육과정 등 지역교육 혁신 관련 정보를 관리합니다. 구체적인 기능 범위는 추후 협의 예정입니다.

#### (8) 융합연구 그룹 — 우선순위 8

4개 융합연구 그룹 관련 정보를 관리합니다. 구체적인 기능 범위는 추후 협의 예정입니다.

### 3.2 제외 범위 (Out of Scope)

아래 기능들은 인터뷰 결과 1차 구축 범위에서 의도적으로 제외되었습니다. 제외 사유를 함께 기술합니다.

| 기능 | 제외 사유 |
|------|----------|
| **봉사활동 관리 / VMS 연동** | 운영 업무 간소화 방침. 봉사활동 인증(VMS/1365)을 플랫폼에서 관리하면 운영자의 업무량이 크게 증가하므로 제외 |
| **ESG/SDGs 성과 관리** | 운영 업무 간소화 방침. 실행계획서에 데이터만 필요하고, 플랫폼 내에서 별도의 성과 관리 기능은 불필요 |
| **개인별 KPI** | 개인정보 보호 이슈. 개인을 특정할 수 있는 성과 지표는 관리하지 않음 |
| **프로젝트 모니터링/체크포인트** | 업무 복잡도 증가 방지. 수행 팀 내부적으로 프로젝트를 관리하고, 플랫폼은 결과물 위주로 관리. 중간 체크포인트를 넣으면 관리자와 참여자 모두의 부담이 증가 |
| **온라인 협업 도구 (아이디어 보드)** | 불필요. 프로젝트 성과 결과물을 업로드하는 수준으로 충분하며, 별도의 브레인스토밍/투표 보드는 필요하지 않음 |
| **팀 단위 권한 관리** | 프로젝트 모니터링이 배제됨에 따라 팀별로 세분화된 권한 관리도 불필요. 별도 관리 포인트를 두면 운영 업무 복잡도가 증가 |
| **글로벌 사업 관리** | 범위 축소. 글로벌 확산은 성공 사례를 외부에 공개하는 소개 페이지 수준이면 충분 |
| **AI 챗봇** | 차년도 별도 발주로 진행. 1차 구축에서는 플랫폼 핵심 기능 안정화에 집중 |

### 3.3 차년도 도입 검토 사항

인터뷰에서 논의된 차년도 이후 도입 검토 기능입니다.

#### AI 챗봇 (국가대표 AI)

인터뷰에서 약 15분간 논의된 주요 토픽입니다. "AI를 넣으면 홈페이지가 예뻐지고, 챗봇 서비스를 넣으면 심사에서 좋은 인상을 줄 수 있다"는 의견이 있었으나, 1차 구축에서는 플랫폼 안정화를 우선하기로 하였습니다.

- **도입 시점**: 2차년도 별도 발주
- **방향**: 국가대표 AI를 활용하면 홍보 효과와 비용 절감 효과가 있음
- **전제조건**: 학교 내 GPU 서버 유무 확인, AI가 참조할 데이터 갱신 파이프라인 구성 필요
- **예산**: +1,000~1,500만원 규모

#### 학교 통합 로그인 (SSO)

현재 학교 포털 아이디로 여러 시스템에 로그인하는 통합 로그인을 사용하고 있으나, 1차 구축에서는 **독자 로그인 체계**로 진행합니다. 4개 캠퍼스(대전/공주/예산/세종) 통합 로그인 여부를 포함하여 추후 검토합니다.

## 4. 콘텐츠 등록 권한

플랫폼 내 각 콘텐츠 유형별로 누가 등록/수정할 수 있는지를 정리합니다.

| 콘텐츠 유형 | 등록 권한 | 설명 |
|-----------|----------|------|
| 지역 문제 (이슈) | 관리자만 | 센터 담당자가 조사 후 직접 등록 |
| 리빙랩 프로젝트 | 관리자만 | 프로젝트 생성, 팀 구성, 상태 관리 모두 관리자 |
| 산출물 (결과물) | 팀 참여자 | 양식을 다운로드받아 작성 후 업로드. 관리자도 등록 가능 |
| 문서 양식 (템플릿) | 관리자만 | 보고서 양식, 활동 기록 양식 등 |
| 소개 페이지 (편집) | 관리자만 | WYSIWYG 에디터로 직접 편집 |
| 성공사례 | 관리자만 | 프로젝트 성과를 정리하여 공개 |
